2. Setup
From the repo root:

pip install -r requirements.txt

3. Core production pipeline
Run these in order for the main project flow:

python ingest.py
python build_features.py
python train_model.py
python src/features/calculate_scheme_deltas.py

What each step does:
- `ingest.py`
  - pulls live NBA API data into SQLite
  - stores raw JSON snapshots under `data/raw/`
- `build_features.py`
  - creates `data/processed/lineup_training_dataset.csv`
- `train_model.py`
  - trains the production XGBoost model
  - writes metrics, feature importance, predictions, and diagnostic plots
- `src/features/calculate_scheme_deltas.py`
  - generates `data/processed/dynamic_scheme_profiles.json`

8. Output reference
Typical demo output fields:
- `Baseline prediction`
- `Actual target` or `Estimated historical target`
- `Recommended scheme`
- `Scheme ranking`
- `Top explanation rows`
- `Source`

Definitions:
- `Baseline prediction`
  - the model's predicted defensive outcome for the lineup
- `Actual target (API defensive rating)`
  - the true historical lineup defensive rating for an exact historical 5-man match
- `Estimated historical target (weighted overlap)`
  - a weighted estimate from similar historical lineups when the exact 5-man lineup does not exist
- `Actual target (fallback per 48)`
  - a backup target only if the advanced lineup endpoint did not provide a usable defensive rating
- `Source`
  - tells you whether the lineup came from an exact historical match, a weighted historical overlap estimate, or a synthetic fallback

9. What the project is and is not
This project is:
- a lineup-aware defensive scheme simulator and recommendation engine

This project is not:
- a supervised defensive scheme classifier

Reason:
- the public NBA API does not provide possession-level `Drop`, `Switch`, or `Zone` labels

10. What changed during the project
Important pivots:
- We initially used the wrong lineup endpoint view and had to switch to the advanced `LeagueDashLineups` view to obtain real `DEF_RATING` and `POSS`.
- We originally relied too much on synthetic custom lineups and later changed the demo to prefer exact historical matches and weighted historical overlap.
- We tested broader feature families because the baseline model relied heavily on player defensive rating aggregates.
- The experiments showed that defensive rating aggregates still remained the strongest valid predictive signal.

12. Common issues
Issue: NBA API blocks requests or times out
Fix:
- rerun later
- keep the built-in `time.sleep()` delays
- do not remove backoff logic

Issue: a player name is not found
Fix:
- use the player's full NBA API name
- try the historical demo mode if the custom lineup is purely hypothetical

Issue: output looks too noisy
Fix:
- inspect the `Source` field
- exact historical lineups are the strongest evidence
- weighted overlap is weaker
- synthetic fallback is the weakest

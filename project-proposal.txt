Project Proposal: Dynamic Lineup-Dependent Defensive Scheme Optimizer
By Gabriele Lisci (gl618) and Victor Olivo (vmo32)

1. Problem Definition and Strategic Aspects

Our project addresses the limitations of static defensive game-planning in the NBA. Coaching staffs often rely on overall team averages or informal judgment when choosing a coverage such as `Switch`, `Drop`, or `Zone`, without explicitly modeling the specific 5-man lineup on the floor. The practical problem is a lack of fast, data-driven support for recommending a defensively sound coverage for the exact lineup currently playing.

The project combines database design, feature engineering, predictive modeling, and explainability into one end-to-end system. We aggregate player-level defensive play-type data and lineup-level defensive outcomes in a relational database, engineer lineup features, and feed those features into a machine-learning-assisted simulation and recommendation engine.

2. Novelty, Importance, and Prior Work

The project is important because modern NBA offenses punish weak links and schematic mismatches quickly. A lineup-aware system that estimates defensive risk and compares alternative coverages can support better tactical choices.

Existing public basketball analytics work is mostly descriptive or predictive at the game and player level. Public models often focus on player value, win probability, or general team performance. They do not usually provide an open, lineup-specific recommendation layer for in-game defensive coverage.

Our contribution is not a possession-level scheme classifier. The public NBA API does not expose labels that say a possession was defended in `Drop`, `Switch`, or `Zone`. Because of that, the project is framed more honestly as a lineup-aware defensive scheme simulator and recommendation engine. It predicts lineup defensive outcome from real data, then simulates how different candidate coverages may change that outcome.

3. Relation to Course Material

The project maps closely to the course pipeline:

- API data collection with `nba_api`
- JSON provenance and data cleaning with Pandas
- relational schema design and SQL-style joins
- feature engineering for model-ready tabular data
- tree-based machine learning using XGBoost
- explainability with SHAP

The raw NBA API responses are stored for reproducibility, normalized into SQLite through SQLAlchemy, joined into lineup-level training rows, and then used in a baseline predictive model.

4. Data Description and Accessibility

Data source:

- public NBA statistics endpoints accessed through the open-source `nba_api` client

Raw format:

- JSON responses from `stats.nba.com`

Stored format:

- normalized SQLite tables plus processed Pandas DataFrames and CSV exports

Primary feature groups:

- player defensive play-type PPP allowed for:
  - `Isolation`
  - `Pick and Roll Ball Handler`
  - `Pick and Roll Roll Man`
  - `Spot Up`
- lineup composition features:
  - guard, forward, and center counts
  - average size measurements
- lineup-level historical outcomes from the advanced lineup endpoint:
  - `DEF_RATING`
  - `PACE`
  - `POSS`

Primary target:

- lineup `DEF_RATING`

Important limitation:

- the public NBA API does not provide true possession-level defensive scheme labels
- because of that, the project cannot be trained as a supervised classifier of `Drop`, `Switch`, or `Zone`

5. Methodology and Implementation Plan

Database setup:

- build a relational schema linking players, defensive play types, lineups, and lineup membership through a many-to-many association table

Ingestion:

- fetch player defense, Synergy defensive play types, and five-man lineup data
- store raw snapshots under `data/raw/`
- normalize the data into SQLite

Feature engineering:

- aggregate five-player defensive profiles into lineup-level rows
- include size, position counts, and play-type PPP summaries
- filter out unstable low-minute lineups and low-possession play-type samples

Model selection:

- use XGBoost as the baseline regressor because it handles nonlinear interactions well in tabular data

Simulation layer:

- score `Drop`, `Switch`, and `Zone` by adjusting lineup features through explicit scheme profiles
- use data-driven dynamic scheme profiles when available, and hardcoded fallback profiles otherwise

Player-input recommendation flow:

1. user enters five players
2. look for an exact historical lineup match
3. if no exact match exists, estimate from weighted historical overlap
4. if overlap evidence is weak, fall back to a synthetic player-profile lineup
5. predict baseline defensive outcome
6. rescore the lineup under each scheme
7. recommend the lowest predicted defensive-cost option

Explainability:

- use SHAP values and explicit scheme adjustments to explain why one scheme was preferred over another

6. Evaluation Metrics and Success Criteria

Baseline predictive accuracy:

- `MAE`
- `RMSE`

Recommendation success:

- compare the simulated predicted defensive outcome for `Drop`, `Switch`, and `Zone`
- success means the engine can return an interpretable ranked recommendation that is grounded in real lineup and play-type data

Project success criteria:

- live NBA ingestion works reliably enough to populate the database
- the feature pipeline produces stable lineup rows
- the baseline model predicts lineup defensive outcome reasonably well
- the recommendation engine returns transparent, explainable coverage rankings for user-specified lineups

7. Current Positioning Statement

The correct description of the project is:

- a lineup-aware defensive scheme simulator and recommendation engine

It should not be described as:

- a supervised defensive scheme classifier

That distinction is central to the integrity of the project because of the public-data labeling limitation.
